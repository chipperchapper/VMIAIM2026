How your question becomes an answer

A plain-English walkthrough of what happens between typing "Who got the most Navy money?" and getting a number back — and why the system can't wreck the data or the budget along the way.

The four players

Everything in this project is one of these four things talking to the others.

🧑 You browser

Ask questions in plain English at localhost:8000 (later: a public URL).

🧠 The brain gemini

A language model (Gemini) that reads your question, plans, and writes SQL. It can only think and talk — it can't touch anything directly.

🔧 The tools our python

Three functions we wrote: list_tables, show_schema, run_query. The only hands the brain has — and every hand wears a safety glove.

🗄️ BigQuery warehouse

Google's database holding the DoD contract data. Does the heavy number-crunching and sends back small result tables.

Think of it as a smart analyst coworker: the brain is their reasoning, the tools are their keyboard, BigQuery is the filing room — and we only gave them a read-only badge for the filing room.

The journey of one question

Numbered because it really happens in this order — this is the loop the ADK (Agent Development Kit) runs every time you hit Enter.

1
you

You ask in English

"Which parent company received the most obligations?"
2
brain · gemini

The brain reads its briefing packet

Before your question even arrives, the brain has been handed a system instruction we build from the semantics/ YAML files: what every table and column means, that "Army" really means Department of the Army, that one contract = many transaction rows, and the approved metric formulas. This packet is why it writes sensible SQL instead of guessing.

3
brain · gemini

It writes one SQL query

SELECT recipient_parent_name, SUM(federal_action_obligation) AS total FROM aim_raw.dod_contracts_test GROUP BY recipient_parent_name ORDER BY total DESC LIMIT 3

SQL is the language databases speak — "group the rows by parent company, add up the dollars, give me the biggest three."

4
tools · our python

It hands the SQL to run_query — and loses control

From here on, our code is in charge, not the model. The query must pass every safety gate below before BigQuery ever sees it.

5
bigquery

BigQuery runs it — with a leash on

Hard caps ride along with the query: scan at most 1 GB, run at most 30 seconds, return at most 100 rows. The database itself also enforces that our identity is read-only — even a bug in our code couldn't change or delete data.

6
tools · our python

Results come back, and the receipt is filed

The rows return to the tool, which writes an audit line (what SQL, how many bytes, how long) to logs/audit.jsonl — so there's always a paper trail.

7
brain · gemini

The brain turns rows into an answer

Top parent company: PARAGON PROFESSIONAL SERVICES LLC ($12.8M)... Method: summed obligations, grouped by parent, negatives included. SQL: [shown]

The "answer contract" in its instructions requires: lead with the answer, show evidence, state the method, show the SQL. If the data can't answer the question, it must say so — not invent a number.

The safety gates

Step 4 in detail. Every query walks this gauntlet — fail any gate and it's rejected with an explanation, never executed. The model can't skip a gate, because the gates live in our Python, not in its prompt.

⏱️
Rate limit Max 12 queries per minute — a runaway loop can't hammer the warehouse.
else: rejected
▼ passed
📖
Read-only check Must be a single SELECT. INSERT, DELETE, DROP, CREATE… are blocked as words — even hidden mid-query.
else: rejected
▼ passed
🗂️
Dataset allowlist May only touch aim_raw, aim_core, aim_analytics. Any other table name → blocked.
else: rejected

Dry run (free) BigQuery pre-checks the syntax and estimates the cost in bytes — before spending anything.
else: rejected

Cost cap + timeout + row cap Executes with maximum-bytes-billed = 1 GB, 30 s timeout, 100-row result limit.
else: killed

Where's the API key? (There isn't one)

Most AI apps carry a secret key that can leak. Ours borrows identity instead — the same way the bq command does.

Your Google sign-inThe browser approvals you did during setup stored a token on this machine ("this laptop acts as you").
→
Vertex AIGoogle Cloud's door to Gemini, inside project vmi-aim-2026. Checks: is this identity allowed here?
→
Gemini answersUsage billed to the project — visible under the sponsor's $25 budget alert. No key to store, rotate, or leak.
🔒 When deployed to Cloud Run, the app swaps "you" for its own service account — an identity that can only run read-only BigQuery jobs and call the model. The empty OPENAI_API_KEY secret from the setup guide stays unused.

The two-position safety switch

One environment variable (SAFETY_SWITCH in .env) decides whether queries actually run.

DRY_RUNdefault
  • Query is validated and cost-estimated…
  • …but never executed
  • Agent reports the SQL it would run
  • Perfect for testing prompts at zero query cost
LIVEexplicit opt-in
  • Requires two variables set on purpose (SAFETY_SWITCH=LIVE + REQUIRE_EXPLICIT_LIVE=true)
  • Queries execute — with every gate still active
  • Real numbers, real (tiny) costs
